# AGENTS.md — Codex 자율 개발·검토 루프 (제14회 산업통상자원부 공공데이터 공모전)

> **이 파일은 OpenAI Codex CLI 가 자동 로드하는 진입점**이다(작업 디렉터리의 AGENTS.md 자동 컨텍스트). Codex 는 이 파일을 읽고 **각 아이디어 프로젝트를 데모앱까지 완성하고 검토**한다.
> **규칙의 단일 출처는 레포 루트 [`CLAUDE.md`](CLAUDE.md)** — Codex 는 시작 시 `CLAUDE.md` 를 직접 열어 읽고 그 규약을 최우선으로 따른다. 본 파일은 운영 루프만 담는다(규칙은 CLAUDE.md, 충돌 시 CLAUDE.md 우선).

## 0. 정체성·시작 절차
1. `CLAUDE.md` 를 읽는다(디렉터리 규약 §1, 산출물 규약 §2, 평가 채점표 §2.1-E, 제출 양식 §2.1-F, 개발 규약 §3, 작업 흐름 §5, DoD §6, 금지사항 §7).
2. **개발 게이트 = 열림(사용자 개발 지시, 2026-06-28).** CLAUDE.md §5 의 [개발 단계]가 활성화됐다 → 각 프로젝트를 개발계획서→과업지시서→`projects/` 데모앱→개발결과보고서→버전 고정까지 끌어올린다.
3. `planning/` 의 조사·추천 문서와 **검증된 데이터 화이트리스트**(§4)를 참조한다.

## 1. 목표
레포 루트의 **각 `NN-<아이디어>/` 프로젝트**(01~45)를, 공모전 발표평가 시연이 가능한 **데모앱 v1**까지 완성하고 **적대적 검토**로 품질을 보증한다. 한 프로젝트가 끝나면 다음으로 이동, 모든 프로젝트가 DoD 를 충족할 때까지 반복.

## 2. 자율 루프 (ASSESS → PLAN → EXECUTE → VERIFY → REPEAT)
1. **ASSESS** — `ls -d [0-9]*/` 로 프로젝트를 훑고, 각 프로젝트의 충족/미달을 §3 DoD 로 실측 표로 만든다(이미 있다고 가정하지 말고 처음 관점에서 재검증 — CLAUDE.md §9.3 전면 재검토).
2. **PLAN** — 미달 항목이 가장 적어 빨리 끝낼 수 있는 프로젝트 또는 우선순위(§7) 순으로 1개를 택해 작업 단위를 정한다.
3. **EXECUTE** — 그 프로젝트를 §3 순서로 완성한다(문서→앱→캡처→보고서). 한 번에 한 프로젝트에 집중(파일 충돌 방지).
4. **VERIFY** — §3 DoD 와 §5 검토 기준을 **실측 명령(§6)** 으로 확인. 미달이면 같은 프로젝트를 보강(새 프로젝트로 넘어가지 않는다).
5. **REPEAT** — 그 프로젝트가 DoD 충족 → 커밋 + 버전 고정(§3.7) → 다음 프로젝트로. 전부 충족 시 종료(`DONE`).

> **조기 종료 금지(CLAUDE.md §9.3).** 게이트 미달 상태로 "나중에 계속"·요약·계획만 내고 멈추지 않는다. 게이트 ✅ 는 실측 수치가 있어야 한다. 토큰을 아끼지 말고 깊게.

## 3. 프로젝트별 완성 DoD (각 `NN-*/` 마다)
순서대로 채운다. (B = 데모 데이터는 키 없이 mock, CLAUDE.md §3.1·§3.4)
1. `biz/2_개발계획서.md` — 목표 한 줄 + WBS(자체/외주 구분) + 기술스택 + 현재상태 (CLAUDE.md §2.2).
2. `biz/3_과업지시서_v1.md` — 5개 섹션(개요·세부내용·수행체계·추진일정·성과품) (CLAUDE.md §2.3).
3. `projects/<app>/v1.html` — **단일 자체완결 HTML**(빌드 불필요·오프라인·`file://` 단독구동), 키 없이 mock 동작. **실 동작 기능 100+**(토스트 mock 금지, 코드 말미 `<!-- 기능 목록: 1..100+ -->` 주석으로 입증), **뷰 8+**, **반응형**(모바일 390px 바텀탭/드로어 + PC 1280px 사이드바, 가로 overflow 0). (CLAUDE.md §3.3)
   - 데이터는 그 프로젝트 `1_제안서.md` §3 의 **검증된 산업부 데이터셋(§4 화이트리스트)** 응답구조를 모사하고, 코드 상단 주석에 "실제로는 data.go.kr <ID> 연동, 키 부재로 mock"을 정직 명시(CLAUDE.md §2.6).
4. `biz/captures/<v1>/`(PC 1280) + `biz/captures/mobile/<v1>/`(390×844) — **각 6장+**, 실제 구동 화면만(목업 금지). **스크린샷을 직접 열어** 사이드바 하이재킹·잘림·빈 차트 없음을 눈으로 확인(CLAUDE.md §3.3 반응형 검수).
5. `biz/3_1_개발결과보고서_v1.md` — 성과품↔납품 매핑표 + 구현범위(기능 목록표 100+) + 환경 + 실행방법 + 캡처 임베드(PC·모바일) + 검수 충족표 + 검토 체크리스트. v1 은 "미흡/잔여" 섹션 쓰지 않음(CLAUDE.md §2.4).
6. **검토(§5)** 통과.
7. **버전 고정** — 검수 통과 커밋을 `release/<project>/v1` 브랜치 또는 `<project>-v1` 태그로 고정(CLAUDE.md §3.5). `v1.html` 파일 보존(덮어쓰기 금지).

## 5. 검토 (적대적 — 각 프로젝트 완성 후)

- 미달이면 "완료" 보고 금지 → 해당 항목 재작업.

## 6. 게이트 실측 명령 (증거 없이 ✅ 금지)
```
ls -d [0-9]*/                                            # 프로젝트 목록
for d in [0-9]*/; do echo "$d $(ls "$d"biz/ 2>/dev/null)"; done   # 문서 존재
grep -c '기능 목록' <proj>/projects/*/v1.html            # 기능 입증 주석
ls <proj>/biz/captures/**/*.png | wc -l                  # 캡처 수
# 화이트리스트 밖 ID 적발:
grep -rhoE 'data\.go\.kr/data/[0-9]+' <proj>/ | grep -oE '[0-9]+' | sort -u   # → §4 와 대조
```
