## 1. 디렉터리 구조 (절대 변경 금지)

워크스페이스 루트에는 참고 PDF·CLAUDE.md 외 잡파일을 두지 않는다. 각 프로젝트는 워크스페이스 루트 바로 아래에 **자체 디렉터리**를 가진다.

```
k_startup_spare/
├── CLAUDE.md                      ← 본 지침
├── ref-*.pdf                      ← 공통 참고자료 (불변)
│
├── <project-name-1>/              ← 프로젝트 1 (예: 2026-실전창업-동아리A)
│   ├── biz/                       ← 비즈니스·기획 산출물 (Markdown)
│   │   ├── 1_제안서.md
│   │   ├── 2_개발계획서.md          ← 전체 개발 계획 (자체 + 외주 통합)
│   │   ├── 3_과업지시서_v1.md       ← 외주 위탁 분 (개발계획서 기반 작성)
│   │   ├── 3_1_개발결과보고서_v1.md
│   │   ├── 4_과업지시서_v2.md
│   │   ├── 4_1_개발결과보고서_v2.md
│   │   ├── 5_research/            ← 조사·근거자료 모음
│   │   │   ├── README.md          ← 자료 목록·요약
│   │   │   └── *.md / *.pdf
│   │   └── captures/              ← 개발결과보고서용 실 구동 캡처
│   │
│   └── projects/                  ← 실제 개발 결과물 (코드)
│       └── <subapp>/              ← 단일 데모 앱 (Next.js, Streamlit 등)
│
├── <project-name-2>/
│   ├── biz/
│   └── projects/
│
└── …
```

규칙
- 새 프로젝트 추가 시 위 구조를 그대로 복제한다.
- 프로젝트명은 `kebab-case` (예: `2026-실전창업-동아리A`). 한글 허용하되 공백·특수문자 지양.
- `biz/` 외부에 문서 산재 금지, `projects/` 외부에 코드 산재 금지.
- 참고 PDF는 워크스페이스 루트의 `ref-*.pdf` 만 공통 사용. 프로젝트별 추가 참고자료는 `biz/5_research/` 안에 둔다.

## 2. `biz/` 산출물 규약

각 문서는 **버전 번호가 파일명 접두사**로 들어간다 (정렬·식별 용이). 한 프로젝트는 최소 1번 (`v1`) 사이클을 완수하고, 피드백·재계약 발생 시 `v2`, `v3`… 로 추가한다.

### 2.0 공통

- **언어**: 한국어 (전문 용어는 원문 병기 허용)
- **포맷**: Markdown (`.md`)만. hwpx/PDF 변환은 최종 제출 단계에서만.
- **그림자료(다이어그램·도표·차트) = 논문 형식·흑백 (사용자 지시, 2026-06-16).** `biz/` **문서에 들어가는 모든 그림자료**(Mermaid 다이어그램, 표, 차트, 캡션 도식 등)는 **학술 논문 figure 스타일**로 만든다.
  - **순수 흑백(검정·흰색 2색만) — 논문 그림처럼 (사용자 지시, 2026-06-18).** **흰 배경(`#ffffff`) + 검은 선·글자·테두리(`#000000`)** 만. **회색 채움·회색 배경·명암 음영 금지**, 컬러 금지(채도색·배경색·하이라이트 금지). 도형은 **흰 바탕에 검은 윤곽선**(line art).
    - ⚠️ Mermaid `theme:neutral` 은 **회색 박스 채움**을 만들어 "순수 흑백"이 아니다 → **금지**. 아래 검증된 init 을 쓴다.
  - 모든 figure 에 **번호·캡션**을 단다(예: `**그림 1.** 시스템 아키텍처`, `**표 2.** 경쟁사 비교`). 본문에서 "그림 N/표 N"으로 인용.
  - **Mermaid 표준 init (이 한 줄을 도식 첫 줄에 그대로)** — base 테마 + 전 변수 흰/검 강제(렌더 검증됨, 회색 0):
    ```
    %%{init: {'theme':'base','themeVariables':{'background':'#ffffff','primaryColor':'#ffffff','primaryBorderColor':'#000000','primaryTextColor':'#000000','secondaryColor':'#ffffff','secondaryBorderColor':'#000000','secondaryTextColor':'#000000','tertiaryColor':'#ffffff','tertiaryBorderColor':'#000000','tertiaryTextColor':'#000000','mainBkg':'#ffffff','secondBkg':'#ffffff','lineColor':'#000000','textColor':'#000000','clusterBkg':'#ffffff','clusterBorder':'#000000','edgeLabelBackground':'#ffffff','actorBkg':'#ffffff','actorBorder':'#000000','noteBkgColor':'#ffffff','noteBorderColor':'#000000','signalColor':'#000000','signalTextColor':'#000000','fontFamily':'Georgia, serif'}}}%%
    ```
    색 지정(`fill:#...` 컬러)·`style`·컬러 `classDef` 금지(흰/검만).
  - 차트(앱 캡처가 아닌 **문서용** 통계 그래프)는 **흰 배경·검은 막대/선·해칭 패턴**으로. 색·회색 음영으로 계열을 구분하지 말고 **해칭 패턴·라벨·테두리**로 구분.
  - **렌더 눈검수 의무**: 그림은 실제로 렌더(`mmdc -i fig.mmd -o fig.png -b white`)해 PNG 를 **직접 열어** 회색·컬러가 없는 순수 흑백인지 확인한다(앱 캡처의 반응형 눈검수와 동일 원칙).
  - ⚠️ **mindmap 은 이 init 을 따르지 않는다(회색 노드) → `flowchart` 로 변환**한다. timeline·gantt·quadrantChart·pie 까지 커버하는 **전 타입 정준 init·타입별 주의·일괄 변환 절차**는 별도 규약 [`그림자료_규약.md`](그림자료_규약.md) 에 통합돼 있다(단일 출처).
  - ⚠️ 이 규칙은 **문서(`biz/`)의 그림자료에만** 적용된다. **데모 앱(`projects/`) UI 는 예외** — 앱은 [§3.3](#33-빌드--데모) 디자인 가이드(컬러)를 따른다. (문서=논문 흑백, 앱=디자인가이드 컬러.)
- **링크**: 상대 경로
- **이모지**: 섹션 식별·상태 마커 외 남용 금지. (논문형 그림자료 안에는 이모지 금지.)
- **표 정렬**: 헤더에 `:---` / `:---:` / `---:` 명시
- **헤더**: 모든 문서 최상단에 `last_updated: YYYY-MM-DD HH:MM` 기재

### 2.1 `1_제안서.md`

- 공고/발주처가 지정한 **섹션 순서를 고정**한다 (변경·추가·삭제 금지).
- 일반 구조: PSST(Problem · Solution · Scale-up · Team) 또는 공고 지정 6개 섹션.
  → **Team 섹션은 본문/표 골격만 두고 내용은 공란**으로 둔다 ([§2.7](#27-행정-양식-처리--팀--서명-정보) 참조).
- 머리표에 프로젝트 메타(사업명·주관기관·트랙·일정·타깃)를 표로 명시.
- 참고 PDF([`../../ref-business-plan-example.pdf`](../../ref-business-plan-example.pdf))는 **목차·표 구조·서술 톤만 참고**. 도메인 어휘(예: "자성분말", "전기강판")를 차용 금지.
- **경영혁신·창업학적 관점 섹션 필수** — 본문에 `## 경영혁신·창업학적 프레임워크` (또는 동등한 이름) 섹션을 두고, 이 사업이 어떤 학술·경영 이론으로 설명되는지 명시한다. 단순 아이템 설명에 그치지 않고 **왜 지금이 기회인지를 이론적으로 정당화**한다.
  - 활용 가능한 프레임워크 예: Christensen 파괴적 혁신 / Kim·Mauborgne 블루오션 / Osterwalder 비즈니스 모델 캔버스 (BMC) / Ries 린 스타트업 / Blank 고객 개발 / Sarasvathy 이펙츄에이션 / JTBD (Jobs To Be Done) / Schumpeter 창조적 파괴 / Porter 5 Forces
  - 최소 1개 프레임워크를 명시적으로 인용하고, 본 사업이 그 프레임워크의 어느 위치/단계에 해당하는지 서술.
- **고객확보(GTM)·수익모델·차별성 섹션 필수 (사용자 지시, 2026-06-10).** 단순 아이템 설명에 그치지 않고 아래 셋을 **구체적 수치·채널·전술**로 명시한다. 추상적 구호("바이럴로 확산한다", "수수료로 번다") 금지 — 채널·숫자·근거를 댄다.
  - **① 고객확보 (Go-to-Market)** — 타깃 고객 세분화(ICP), 획득 채널(유료/오가닉/제휴)별 전술, 인지→가입→활성→유지 퍼널, 초기 트랙션 확보 계획(첫 100·1,000 사용자 확보 방법), 예상 **CAC**, 리텐션 가설.
  - **② 수익모델** — 수익원(구독·거래수수료·광고·B2B 라이선스 등)과 **가격 정책**, **단위경제성(LTV·CAC·LTV/CAC·회수기간·기여이익)**, 매출 시나리오(보수/기본/공격 3안).
  - **③ 차별성·경쟁우위(Moat)** — 직접·간접 경쟁자 비교표 + 차별점, 방어가능성(데이터·네트워크 효과·전환비용·IP·규제 해자), **Why us / Why now**.
  - 가능하면 §경영혁신·창업학적 프레임워크와 연결한다(예: 블루오션=차별성, 린스타트업=고객확보 실험, BMC=수익모델).
- **차별점 최소 50개 도출 (사용자 지시, 2026-06-16).** 차별성 섹션에는 경쟁사 대비 차별점을 **최소 50개 이상** 구조적으로 도출·정리한다(`**표 N.** 차별점 도출` 형태의 흑백 표 권장).
  - 단순 나열이 아니라 **카테고리로 묶어** 제시(예: 기술·데이터·UX·가격·GTM·규제·네트워크효과·운영 등 축별로 N개씩) → 합계 50+.
  - 각 차별점은 *경쟁사 현황 → 우리 차별점 → 고객 가치(가능하면 수치)* 형태로. 50개 중 핵심 것들은 [§차별화 기술의 구매동인 논증](#21-1_제안서md)으로 연결해 *실제 구매동인인지* 검증한다.
  - 50개를 채우려고 **사소·중복·억지 항목으로 부풀리지 말 것** — 의미 있는 차별점이 그만큼 안 나오면 시장·경쟁사 조사를 `5_research/`로 더 넓혀 근거 있는 항목으로 채운다([§2.6](#26-데이터-정직성-제안서과업지시서결과보고서-공통) 정직성 우선).
- **차별화 기술의 "구매동인" 논증 필수 (사용자 지시, 2026-06-16).** 경쟁사와 다른 기술·기능을 **나열**하는 데서 그치지 말고, 그 차별점이 **고객의 실제 구매 결정·사용 동기를 얼마나 크게 움직이는지를 조사하고 논증**한다. "우리가 더 빠르다/정확하다"는 자기주장이 아니라, *그래서 고객이 돈을 내거나 매일 쓰게 되는가*를 근거로 증명한다. 본문에 `## 차별화 기술의 구매동인 논증`(또는 §차별성 하위 단락)을 두고 아래를 채운다.
  - **① 구매동인 가설 명시** — 본 기술 차별점이 건드리는 고객의 핵심 의사결정 요인(JTBD 의 "더 나은 채용 기준"). 그 요인이 *must-have(미충족 시 구매·사용 안 함)* 인지 *nice-to-have(있으면 좋음)* 인지 분류하고 근거를 댄다.
  - **② 크기 정량화** — 차별점이 만드는 가치를 **고객 언어의 수치**로 환산(예: 시간 −N시간/월, 비용 −N원/건, 오류·리스크 −N%, 매출 +N%, 회수기간 N개월). 이 수치가 고객의 **전환비용·기존 대안 대비 충분히 큰지**(예: 전환 마찰을 넘는 10배 규칙 등)를 따진다.
  - **③ 외부 근거로 뒷받침** — 위 주장을 **`5_research/` 의 출처**(시장조사·고객 인터뷰·산업 벤치마크·경쟁사 공개자료·유사 제품 도입효과 사례)로 연결한다. 자체 추정은 `[추정]` 표기하고, 검증된 외부 수치와 섞지 않는다. **근거 없는 "고객이 좋아할 것이다"식 단정 금지.**
  - **④ 반증·대안 위협 직시** — 고객이 *그 차별점에도 불구하고 안 사는/이탈하는* 이유(가격 민감도·기존 워크플로 관성·"충분히 좋은" 무료 대안)를 적고, 그 위협을 어떻게 넘는지 답한다. 차별점이 실은 약한 구매동인이면 그렇게 정직히 쓰고 진짜 동인을 찾는다.
  - 가능하면 데모 앱(`projects/`)이 이 구매동인을 **실제로 구현·시연**하는 지점을 본문에서 가리킨다(논증과 산출물 정합).
- **AI 제품 = 'API 래퍼' 금지, 해자 논증 필수 (사용자 지시, 2026-06-25).** AI/LLM/ML 을 쓰는 제품은 **단순히 ChatGPT·외부 LLM API 를 호출해 결과를 보여주는 얇은 래퍼(thin wrapper)면 안 된다.** "프롬프트 입력 → 모델 답 출력"만으로는 해자가 없다(누구나 복제). 제안서 차별성/구매동인에 **모델 호출 위에 쌓은 독자적 가치(why-not-a-wrapper)** 를 반드시 논증한다:
  - **① 독자 자산** — 자체 도메인 데이터/말뭉치, 사용 결과 피드백 루프(데이터 네트워크 효과), 도메인 특화 룰·온톨로지·템플릿, 평가/품질(eval) 레이어, 미세조정·검색증강(RAG) 파이프라인 중 무엇이 모델을 대체불가하게 만드는가.
  - **② 버티컬 워크플로·통합** — 단발 생성이 아니라 *업무 전 과정*에 끼어드는 워크플로, 기존 시스템 연동, 규제/도메인 준수, 휴먼인더루프 검수 등 *모델 밖* 해자.
  - **③ 모델 교체가능성 전제** — 기반 LLM 은 상품화(commoditize)된다고 보고, **모델이 바뀌어도 남는 가치**가 무엇인지 명시. "GPT 가 더 좋아지면 우리도 좋아진다"식 의존이 아니라 우리 고유 레이어로 방어.
  - **④ 데모 정직성** — 데모 앱의 "AI"가 **결정론적 룰/템플릿 mock** 이면 [§2.6](#26-데이터-정직성-제안서과업지시서결과보고서-공통) 대로 *LLM 아님*을 정직히 표기하고, 제안서는 그 mock 이 대체할 **실제 독자 엔진(데이터·워크플로 해자)** 을 가리킨다. 데모가 곧 해자인 척하지 않는다.
  - (래퍼가 아님을 못 밝히면 그 AI 차별점은 약한 구매동인이다 — [§차별점 50+]·구매동인 논증에서 정직히 강등.)
- **참고문헌 최소 1,000개 (사용자 지시, 2026-06-16).** 본문 말미에 **`## 참고문헌`** 섹션을 두고, 각주/인용 출처를 **누적 1,000개 이상** 모은다(`[^1]`…`[^1000]` 형태, `5_research/`와 1:1 연결).
  - **정직성이 수량보다 우선 ([§2.6](#26-데이터-정직성-제안서과업지시서결과보고서-공통)).** 1,000개는 **실제 존재하는 출처**여야 한다 — 날조·중복·유령 인용(존재하지 않는 URL/논문) **절대 금지**. 같은 출처를 쪼개 부풀리지 않는다.
  - 수량 확보 방법: 조사 범위를 넓힌다 — 공공통계 항목별(KOSIS 표 단위)·정부 부처 보도자료·국책연구원 보고서·KCI/DBpia/RISS 논문·국제기구(OECD/ILO/WHO)·특허·표준·경쟁사 공개자료·뉴스·업계 리포트. **출처 수집은 [§9 병렬 에이전트](#9-병렬-에이전트-오케스트레이션-사용자-지시-2026-06-16)로 분산**하면 1,000개 도달이 현실적이다.
  - 진행 중이라 1,000개 미달이면 `## 참고문헌` 머리에 **현재 수량/목표(예: `412 / 1,000`)** 와 미달 사유를 명시(허위 충족 표기 금지).

**외주 용역 기업에 시제품 또는 작업을 맡길 때** 발주자 측이 작성하는 지시서. 개발계획서가 완료된 뒤에 작성하며, 무엇을·어디까지·어떤 조건에서 맡길지를 확정한다.

**작성 순서**: [`2_개발계획서.md`](#22-2_개발계획서md) §3 WBS 의 외주 구분 행 → 본 문서 §1 과업의 범위로 옮김 → 세부 사양·일정·검수 기준 보강.

문서는 **5개 최상위 섹션**으로 구성한다.

#### 1. 과업의 개요

**과업의 목적**
> 외주용역 기업에 시제품 또는 작업을 맡기는 **목적**을 서술한다.

1) 외주에 맡기는 직접 목적 (예: "전기자전거 핸들 제어부 시제품 1대 제작")
2) 그 목적이 발주자 측 사업·제안([`1_제안서.md`](./1_제안서.md))과 어떻게 연결되는가

**과업의 범위**
> 시제품 제작을 위해 **외주 용역 회사에서 진행해야 하는 업무**를 빠짐없이 나열한다.

1) 외주사가 수행할 업무 항목 (설계·부품 수급·가공·조립·시험·납품 등)
2) 범위 외 (Out of Scope) — 외주사가 수행하지 않는 업무 (분쟁 예방)

#### 2. 과업의 세부 내용

> 시제품 또는 홈페이지 등 **업무의 세부적인 내용**을 작성.

- 시제품 사양 (재료·치수·전기·기계·SW 사양)
- 사용자를 위한 UI/UX 제작 사양 (화면 흐름·디자인 가이드 등)
- 사용 부품·라이브러리·외부 의존성
- 품질·성능 기준 (정량 수치)

#### 3. 과업의 수행체계 및 방법

- 외주사 조직·담당자 (PM, 엔지니어, 디자이너 등 R&R)
- 발주자 측 협력 창구 — **이름·연락처·소속은 공란**, 항목 골격만 두고 사용자가 채움
- 작업 방법론 (개발 방식·형상관리·테스트 절차)
- 산출물 인도 방식 (전달 매체·형식·접근 권한)

#### 4. 추진일정

- 단계별 Start·End·Status 표
- 중간 보고·중간 검수 일정 명시
- 최종 납품일·검수일

#### 5. 성과품 목록

- 납품할 산출물 항목별 수량·형태·검수 기준
- 예: 시제품 N대 / 회로도 / 펌웨어 소스 / 사용자 매뉴얼 / 시험성적서 / 디자인 시안 PSD 등
- 각 성과품의 검수 합격 조건을 측정 가능한 형태로 명시

> v2 이상 버전에는 문서 말미에 **변경 이력** 단락(v1→vN 변경점)을 추가한다.
> `v2`는 `v1` 검수 후 추가/변경된 과업이 발생했을 때 생성한다. **v1을 수정하지 않고 새 파일로 분리.**

### 2.4 `3_1_개발결과보고서_v1.md` · `4_1_개발결과보고서_v2.md`

해당 버전 과업지시서의 **검수 증거** 문서. 외주사가 납품한 산출물(또는 자체 개발 결과)이 과업지시서 §5 성과품 목록을 충족하는지 실제 구동 캡처와 함께 검증한다.

#### v1 vs v2 — 결정적 차이

**v1 결과보고서**
- 1차 산출물의 정상 작동을 증거로 보인다.
- "미흡 / 잔여 / 한계점" 섹션은 **쓰지 않는다.** v1에서 한계를 자진 노출하면 검수자가 그 점만 본다.
- §7 자리는 비워두거나 "추가 확장 가능 영역" 처럼 중립 어조로만 짧게.

**v2 결과보고서 (필수 구조)**
- v2는 v1의 **부족함을 메우는 사이클**이다. 다음 두 섹션이 보고서 본문에 반드시 들어간다:
  1. `## v1 한계 및 v2 개선 매핑` — v1에서 부족했던 항목 ↔ v2에서 해결한 방식 1:1 표.
  2. `## v2 신규/심화 산출물` — v2에서 새로 추가되거나 심화한 기능을 캡처와 함께 입증.
- v2 결과보고서 자체는 "v2 한계"를 쓰지 않는다 (v3 시점에서 다룬다).

#### 품질 기준 (가치 환산)

| 사이클 | 최소 가치 기준 | 의미 |
|---|---|---|
| **v1** | **최소 1천만원** | 외주 위탁 시 1천만원(약 시니어 1~2주) 비용을 지불할 만한 PoC. 단순 정적 mock·토스트 위주는 v1 미달 |
| **v2** | **최소 5억** | 베타 출시·시리즈A 데모 가능한 수준. 실 백엔드 시뮬레이션·다중 사용자 개념·모든 핵심 기능 실 구현 |

**v1 (1천만원) 체크 기준 — 모두 충족**
- 모든 핵심 액션이 **실 동작** (토스트 mock 금지). 사진은 실제 업로드·미리보기, 음성은 Web Speech, 서명은 캔버스→이미지 저장, PDF는 jsPDF 실 생성 등.
- **다단계 워크플로** 1개 이상 (예: 선택 → 작성 → 발송 → 추적 4단계).
- **상태 지속성** — localStorage / IndexedDB 등으로 새로고침 후에도 유지.
- 화면/뷰 **4종 이상**, 각 뷰가 단순 표시가 아니라 인터랙션 가능.
- 신규 캡처 **6장 이상** (액션 결과 화면 포함).

**v2 (5억) 체크 기준 — v1 충족 위에 모두 추가**
- v1 한계 매핑표 + 1:1 해결 증거.
- **실 알고리즘** 도입 (예: Markov 어트리뷰션, Shapley value, RAG, 실 이미지 분석 등).
- **다중 사용자/다중 테넌트** 시뮬레이션 (역할별 화면, 권한 분기).
- **외부 시스템 통합** 2건 이상 (CDN·라이브러리는 카운트 아님 — 카카오 알림톡 mock 발송 로그·CSV 외부 입출력·OAuth 흐름 등 실 통합).
- 화면/뷰 **8종 이상**, 워크플로 3개 이상.
- 신규 캡처 **10장 이상**.

기준 미달이면 새 사이클 만들지 말 것 — 같은 사이클 내 패치로 처리한다.

**A. 캡처 의무**
- **실제 구동된 화면** 스크린샷만 사용. 목업·합성 이미지 금지.
- 캡처는 [`captures/`](./captures/) 에 PNG로 저장.

- **모바일·PC 캡처 분리 의무 (사용자 지시, 2026-06-10).** 웹 앱은 **PC와 모바일 양쪽**을 각각 캡처한다. 예: `captures/<vN>/`(PC, 데스크톱 뷰포트 1280±) 와 `captures/mobile/<vN>/`(모바일, 390×844 등) 로 **폴더를 나눠** 저장. 결과보고서엔 양쪽을 모두 임베드한다.

**B. 작성 절차**
1. 코드 빌드·실행 → 실 화면 표시


### 2.9 PDF → Markdown 변환 (참고자료)

참고/양식 PDF 는 [opendataloader-pdf](https://github.com/opendataloader-project/opendataloader-pdf)(Apache-2.0, Java 11+ 필요)로 마크다운 변환해 `.md` companion 을 함께 보관한다(본문 인용·표 구조 참조에 사용). 격리 환경에서 `pip install opendataloader-pdf` 후 `opendataloader_pdf.convert(input_path=[...], output_dir=..., format="markdown")`. **데이터가 채워진 표는 GFM 표로 정상 변환되지만, 셀이 빈 양식 PDF(예: 거래명세서)는 격자선만 있어 표로 인식되지 않으므로 원본 레이아웃을 직접 복원**한다.

## 3. `projects/` 개발 규약

각 프로젝트의 실제 개발 결과물은 `<project-name>/projects/<subapp>/` 에 둔다.

### 3.1 환경변수 / 시크릿
- API 키는 **절대 코드에 하드코딩 금지**.
- `.env` 파일 사용 + `.gitignore` 등록.
- 키 부재 시 **모킹/mock fallback** 으로 동작 (오프라인 데모 가능성 보장).

### 3.2 의존성
- 필요 시에만 추가, 미사용 의존성은 즉시 제거.
- 검증 안 된 라이브러리 무분별 추가 금지.

### 3.3 빌드 & 데모
- 데모 앱은 프로덕션 빌드(`pnpm build` / `python -m build` 등)가 항상 통과해야 한다.
- UI 변경 시 → 실 구동 → 스크린샷 → 해당 버전 개발결과보고서에 반영.
- 단일 데모는 단독 구동 가능해야 한다 (외부 인프라 없이도).
- **기능 최소 100개 (사용자 지시, 2026-06-16).** 데모 앱은 누적 **실 동작 기능 100개 이상**을 구현한다(토스트·정적 mock 은 카운트 제외 — 실제 동작하는 기능만).
  - **기능 = 사용자가 수행 가능한 구체 액션/뷰/연산** (예: "월세 청구서 생성", "연체 스코어 계산", "CSV 내보내기", "역할 전환", "PDF 발행", "차트 필터" 각각 1개). 결과보고서 §2 구현범위에 **기능 목록표**로 번호 매겨 100+ 를 입증한다.
  - 단일 버전에서 100개가 무리면 **버전 누적(v1+v2+v3 합산)으로 100+** 를 달성하되, 각 버전 보고서에 그 시점 누적 수를 명시. 억지 기능 분할로 부풀리지 말 것(의미 있는 기능만).
- **반응형 필수 — 모바일·PC 모두 대응 (사용자 지시, 2026-06-10).** 모든 웹 데모는 **PC(데스크톱)와 모바일 양쪽에서 정상 동작·가독**해야 한다.
  - `<meta name="viewport" content="width=device-width, initial-scale=1">` 필수. **`width=1280/1440` 데스크톱 고정으로 "모바일 대응" 갈음 금지.**
  - 좁은 화면(≈360~430px)에서 **가로 스크롤(overflow) 0**, 텍스트·버튼·표·모달이 깨지지 않게. 넓은 화면에선 콘텐츠 최대 폭(문서형 ≈720px)·여백 위계 유지.
  - **모바일은 PC 축소판이 아니라 레이아웃을 새로 짠다** — 데스크톱 사이드바는 모바일에서 **바텀 탭바 또는 햄버거 오프캔버스 드로어**로(사이드바 그대로 노출·상단 패널 하이재킹 금지), 그리드는 스택, 차트는 고정높이.
  - 디자인 가이드: 톤=[디자인_애플_토스.md](./design-guide/디자인_애플_토스.md) · 모바일=[디자인_모바일.md](./design-guide/디자인_모바일.md) · 데스크톱=[디자인_PC.md](./design-guide/디자인_PC.md).
  - **출고 전 [반응형_검수_필수.md](./design-guide/반응형_검수_필수.md) 통과 의무 (사용자 지시, 2026-06-11).** `overflow=0`·`pageerror=0`은 정상 증거가 **아니다** — **390px·1280px 스크린샷을 직접 눈으로 보고** 사이드바 하이재킹·잘림·빈 차트 등 실패 모드가 없음을 확인해야 "정상"이라 보고한다.
  - 검수 시 **PC·모바일 두 뷰포트에서 각각 실 구동 캡처**([§2.4](#24-3_1_개발결과보고서_v1md--4_1_개발결과보고서_v2md) A).

### 3.4 인증·로그인 정책 (시제품 단계 기본값)

- **로그인이 필요해 보이는 기능도, 우선은 로그인 없이 사용 가능하게 구현**한다.
  시제품·데모 단계에서는 인증이 본질 기능이 아니다. 인증을 먼저 붙이면 캡처·시연·테스트 사이클이 길어진다.
- 권장 패턴
  - 인증 미들웨어는 두되 **기본 사용자(`guest` / `demo-user`)로 자동 통과**시키는 스위치 제공.
  - 사용자별 데이터가 필요하면 **고정 데모 계정**(예: `demo@example.com`)을 시드로 주입.
  - 로그인 화면 자체가 시연 대상이 아니라면 **라우팅 단계에서 우회**.
- 인증이 본질 기능인 경우(로그인 자체가 과업의 산출물)에만 정식 구현. 그 외에는 v2 이후로 미룬다.
- 외부 OAuth·SSO 연동은 키 부재 시 **mock 토큰으로 통과**하게 둔다 (오프라인 시연 보장).

### 3.5 버전별 Git 브랜치 정책 — 언제든 v1/v2 상태 배포 (사용자 지시, 2026-06-16)

> **각 개발 사이클(`vN`)은 그 시점의 앱 상태를 독립적으로 배포·시연할 수 있어야 한다.** 심사·시연 자리에서 "v1은 어땠고 v2에서 무엇이 좋아졌나"를 보여주려면, 과거 버전을 코드째 되살릴 수 있어야 한다. 단일 HTML 자체완결 데모라도 마찬가지다.

규칙
- **버전별 브랜치를 둔다.** 각 사이클 완료 시 그 상태를 `release/<project>/v1`, `release/<project>/v2` … 브랜치(또는 동등한 `v1`/`v2` 태그)로 고정한다. `master`는 항상 최신(최고 vN)을 가리킨다.
  - 예: `git switch -c release/2026-saas-rental-manager/v1` → v1 결과보고서 검수 통과 직후 커밋·고정.
  - 태그를 병행해도 좋다: `git tag <project>-v1 <commit>` (불변 스냅샷).
- **언제든 배포 가능**해야 한다 — 임의 vN 브랜치/태그를 체크아웃하면 그 버전 앱이 **그대로 빌드·구동**된다. v2 작업이 v1 데모를 깨뜨리면 안 된다.
  - 단일 HTML 데모는 `v1.html`·`v2.html`처럼 **버전별 파일을 보존**(덮어쓰기 금지)하므로 브랜치와 함께 파일 단위로도 과거 배포가 보장된다. 브랜치 정책은 그 위에 "프로젝트 전체 상태(문서·캡처 포함) 스냅샷"을 추가로 고정하는 장치다.
- **검수 통과 = 브랜치/태그 고정 시점.** [§2.4](#24-3_1_개발결과보고서_v1md--4_1_개발결과보고서_v2md) v1/v2 결과보고서가 검수를 통과하면, 그 커밋을 해당 버전 브랜치/태그로 고정한 뒤 `master`에서 다음 vN 작업을 잇는다.
- **v1 불변 원칙과 정합.** v2 작업은 `master`(또는 v2 브랜치)에서만 진행하고, 이미 고정한 v1 브랜치/태그·`v1.html`은 건드리지 않는다([§7](#7-절대-하지-말-것) "v1 직접 수정 금지"와 동일 취지).
- 브랜치/태그 네이밍은 위 예시를 표준으로 한다. 실 푸시·배포(원격 반영)는 [Repo 가시성 기본값](#) 등 사용자 정책을 따르되, **로컬에서 vN 브랜치/태그가 존재**하는 것까지는 사이클 완료의 일부다.

## 8. 행동 지침 — 흔한 LLM 코딩 실수 줄이기 (사용자 지시, 2026-06-16)

> Behavioral guidelines to reduce common LLM coding mistakes. Merge with project-specific instructions as needed.
>
> **Tradeoff:** These guidelines bias toward caution over speed. For trivial tasks, use judgment.

### 8.1 Think Before Coding
**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

### 8.2 Simplicity First
**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: *"Would a senior engineer say this is overcomplicated?"* If yes, simplify.

### 8.3 Surgical Changes
**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it — don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

**The test:** Every changed line should trace directly to the user's request.

### 8.4 Goal-Driven Execution
**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

> These guidelines are working if: fewer unnecessary changes in diffs, fewer rewrites due to overcomplication, and clarifying questions come before implementation rather than after mistakes.

## 9. 병렬 에이전트 오케스트레이션 (사용자 지시, 2026-06-16)

> 이 레포는 **재사용 템플릿**이다. 큰 정량 기준(참고문헌 1,000+·차별점 50+)을 한 세션이 직렬로 채우기 어렵다 → **작업을 역할별로 분리해 병렬 에이전트로 분산**한다. **Claude Code 전용**으로 운영한다.

### 9.1 에이전트 정의 위치
- **Claude Code**: [`.claude/agents/`](.claude/agents/) 의 각 `*.md`(YAML frontmatter `name`/`description`/`tools` + 시스템 프롬프트). `Agent`/Task 도구로 `subagent_type: <name>` 지정해 **여러 개를 한 메시지에서 동시 spawn**(병렬). 모든 하위 에이전트는 **본 CLAUDE.md 규칙을 단일 출처**로 따른다.

### 9.2 표준 역할(병렬 단위)
| 에이전트 | 책임 | 산출물 |
|---|---|---|
| `proposal-writer` | 제안서 집필(PSST·프레임워크·GTM/수익/차별성·구매동인) | `biz/1_제안서.md` |
| `research-collector` | 출처 수집(목표 **1,000+**), 정직성 검증 | `biz/5_research/` |
| `differentiation-analyst` | 차별점 **50+** 도출·카테고리화·구매동인 연결 | 제안서 차별성 섹션 |
| `figure-maker` | **순수 흑백 논문형** 그림자료(Mermaid base 흰/검 강제·표·차트, 회색 0) 생성 | 각 문서 figure |
| `app-builder` | 데모 앱(기능 **100+**·반응형·디자인가이드) | `projects/<app>/vN.html` |
| `capture-runner` | 실 구동 캡처(PC+모바일 분리)·반응형 눈검수 | `biz/captures/` |
| `report-writer` | 과업지시서·개발결과보고서 | `biz/N_*_vN.md` |
| `qa-auditor` | 적대적 검수(DoD·정직성·정량기준 충족) | 검수 코멘트 |

### 9.3 병렬 실행 원칙
- **독립 디렉터리/파일로 분리**해 충돌을 막는다(에이전트 간 같은 파일 동시 수정 금지). git 커밋·브랜치/태그 고정은 **오케스트레이터(메인)가 일괄** 처리하고, 하위 에이전트는 파일·캡처만 만든다([§3.5](#35-버전별-git-브랜치-정책--언제든-v1v2-상태-배포-사용자-지시-2026-06-16)).
- 큰 수량 목표(1,000 출처 등)는 **N개 에이전트로 분할 수집 → 중복 제거 → 정직성 검증 → 통합**.
- 모든 하위 에이전트는 **본 CLAUDE.md 규칙을 최우선**으로 따른다. 정량 미달이어도 **허위 충족 표기 금지**([§2.6](#26-데이터-정직성-제안서과업지시서결과보고서-공통)).
- **기획 부트스트랩 (자율 시작)**: 프로젝트가 없으면 **공고 PDF**(루트 또는 `forms/refs/`)를 출발 씨앗으로, 메타를 **공고 사실만** 뽑아 프로젝트 디렉터리·제안서 머리표를 세운 뒤 루프에 진입한다(없으면 한 줄 아이템/키트 예시).
  - **자율 루프 범위 = 제안서 + 조사까지 ([§5 기본 모드](#5-작업-흐름-프로젝트-단위), 사용자 지시 2026-06-27).** 자율/병렬 루프는 **아이디어별 프로젝트 디렉터리 + 제안서 + `5_research/` 조사**를 채우는 데 집중하고, **개발(app-builder·capture-runner·report-writer 역할)은 사용자의 별도 개발 지시 전까지 spawn 하지 않는다.** 즉 위 §9.2 표에서 평시 활성 역할은 `proposal-writer`·`research-collector`·`differentiation-analyst`·`figure-maker`·`qa-auditor` 이고, 나머지 개발 역할은 개발 지시 후 활성화된다.
- **전면 재검토 원칙 (사용자 지시, 2026-06-23)**: 산출물이 "이미 있다"고 가정하지 않는다. 자율 루프는 매 사이클 기존 산출물을 **처음 관점에서 다시 의심·실측**(품질·정합·정직성·정량)해 끌어올린다 — 증분 점검·자기만족 금지.
- **작업 강도·실측 검증 (사용자 지시, 2026-06-24)**: 작업은 방대하다 — **토큰을 아끼지 말고 깊게** 일한다(짧게 끝나면 일을 안 한 것). 게이트 미달 상태로 "나중에 계속"·요약·계획만 내고 **턴을 끝내지 않는다**(조기 종료 금지). 게이트 ✅ 는 **실측 수치**(grep/카운트/렌더)가 있어야 하며, 골격·스텁으로 수량을 위장하지 않는다.
